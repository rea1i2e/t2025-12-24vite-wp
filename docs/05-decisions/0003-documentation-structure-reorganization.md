# 3. ドキュメント構造の再編成

**ステータス**: 承認済み

## コンテキスト

プロジェクトのドキュメントが増加し、以下の問題が発生していました：

1. **読む順番が不明確**: 初めて使う人がどこから読むべきか分からない
2. **役割分担が不明確**: `AGENTS.md` と `docs/architecture.md` に重複がある
3. **テンプレートとしての位置づけが不明確**: 案件固有情報をどこに書くべきか分からない
4. **ドキュメントの階層構造が不明確**: すべてのドキュメントが `docs/` 直下にあり、重要度や読む順番が分からない

テンプレートリポジトリとして、複数案件で再利用されることを前提に、人間（コーダー）とAI（Cursor / ChatGPT）の両方が迷わず使える構成にする必要がありました。

## 決定

ドキュメントを数字プレフィックス（00-06）で階層化し、以下の構成に再編成しました：

```
docs/
├── 00-getting-started/      # 初めて使う人向け（必須）
│   ├── README.md
│   ├── quick-start.md
│   └── setup.md
├── 01-development/           # 開発者向け（日常的に参照）
│   ├── README.md
│   ├── architecture.md
│   ├── coding-standards.md
│   └── development.md
├── 02-deployment/            # デプロイ関連
│   ├── README.md
│   └── deploy.md
├── 03-troubleshooting/       # トラブルシューティング
│   └── troubleshooting.md
├── 04-project-specific/     # 案件固有情報（テンプレート）
│   ├── README.md
│   └── overview.md
├── 05-decisions/             # 意思決定ログ（ADR）
│   ├── README.md
│   └── ...
└── 06-reference/             # 参考資料（任意）
    ├── README.md
    └── migration-log.md
```

### 主な変更点

1. **数字プレフィックスで読む順番を明確化**: 00から順番に読むことで、初めて使う人でも迷わない
2. **各セクションにREADME.mdを追加**: 各セクションの役割とドキュメント一覧を明記
3. **コーディング規約を独立**: `coding-standards.md` として独立させ、`architecture.md` は設計判断に集中
4. **AGENTS.mdの整理**: AI向けの参照用に整理し、詳細は各ドキュメントへ参照
5. **README.mdの改善**: 読む順番ガイドとドキュメントマップを追加

## 結果

この再編成により、以下の効果が得られました：

1. **読む順番が明確**: 初めて使う人は `00-getting-started/` から始められる
2. **役割分担が明確**: 各ドキュメントの役割が明確になり、重複が解消された
3. **テンプレートとして使いやすい**: 案件固有情報は `04-project-specific/` に集約され、テンプレートとしての位置づけが明確になった
4. **人間とAIの両方が使いやすい**: 人間は階層化されたドキュメント構造で迷わず、AIは `AGENTS.md` から適切なドキュメントを参照できる
